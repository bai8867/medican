# 校园药膳推荐系统 · 产品手册

> 本文档描述 **medican** 产品的能力边界、用户角色与使用方式。详细本地启动、数据库与脚本约定见仓库根目录 [`README.md`](../../README.md)，开发与工具链约定见 [`AGENTS.md`](../../AGENTS.md)。

---

## 1. 产品概述

**校园药膳推荐系统**是一套面向校园场景的 **Web 应用**：结合中医体质与季节等因素，为学生提供药膳推荐、周历浏览、场景化食疗方案，并可选接入 **大模型** 生成个性化食疗建议；同时为管理员与食堂运营人员提供内容与用户治理能力。

- **学生端（校园端）**：移动端友好的浏览与交互（Vue 3 + Vant / Element Plus），需登录后使用核心功能。
- **管理后台**：数据看板、药膳/食材维护、用户与系统设置等（权限分角色）。
- **服务端**：Spring Boot 2.7（Java 11）提供 REST API、鉴权与健康检查等。

## 2. 产品定位与价值

| 维度 | 说明 |
|------|------|
| 目标用户 | 在校学生（体质与偏好驱动推荐）；管理员、食堂运营（内容与数据维护） |
| 核心价值 | 将「体质 / 季节 / 场景」与药膳知识结构化，降低学生选择成本；后台统一维护菜谱与运营数据 |
| 差异化能力 | **场景食疗**（按校园生活场景组织方案）、**AI 食疗方案**（依赖配置的上游大模型，可选） |
| 使用形态 | 浏览器访问；本地或内网联调时前端开发端口 **11999**，默认 API 代理至后端 **11888** |

## 3. 入口与访问控制

### 3.1 门户选择页（`/`，「选择模式」）

应用标题为 **「校园药膳推荐系统」**。用户须先在门户页选择进入方式：

- **学生端 · 校园端**：进入校园登录流程（推荐、体质采集、AI 等）。
- **后台管理**：进入管理端登录（需具备相应角色的账号）。

**产品规则**：不可通过直接粘贴或收藏内页、登录页 URL 绕过门户步骤；路由会校验「门户已选择」标记，未选择则重定向回门户。设计意图是明确使用场景并减少误闯后台或跳过引导。

### 3.2 校园端登录（`/campus/login`）

选择学生端后进入。登录成功后签发 **JWT**，后续 API 请求携带令牌。

### 3.3 管理端登录（`/admin/login`）

选择后台管理后进入。会话与角色由管理端状态维护，部分菜单按角色展示。

## 4. 用户角色与权限（摘要）

| 角色 | 典型用途 | 管理端能力摘要 |
|------|----------|----------------|
| **student** | 学生使用推荐、日历、场景、AI、个人设置 | 无后台入口（校园端为主） |
| **canteen** / 食堂相关角色 | 食堂运营维护周历与菜谱相关内容 | 本周药膳日历、药膳管理、食材管理等（无「仅管理员」模块） |
| **admin** | 系统与人员治理 | 含用户管理、系统设置、AI 质量治理等全量后台能力 |

具体页面级权限以后台路由 `adminRoles` 为准；无权限访问时提示并回到看板。

**账号状态**：学生账号若被后台禁用，校园端会引导至「账号不可用」页；本地禁用状态与后端不一致时，会以后端校验为准。

## 5. 学生端（校园端）功能模块

以下路径均为前端路由，便于产品与测试对照。

| 模块 | 路径 | 说明 |
|------|------|------|
| 首页 / 推荐 | `/home` | 基于用户画像的推荐入口；**无本地体质画像时**，进入首页会优先引导至体质采集（若从场景食疗带 `scene_tag` 进入则可先浏览场景相关推荐） |
| 体质采集 | `/constitution` | 问卷或流程采集体质信息，用于后续推荐与 AI 上下文 |
| 本周药膳日历 | `/calendar` | 按周展示药膳安排 |
| 菜谱详情 | `/recipe/:id` | 单条菜谱详情（兼容历史 `?recipe_id=` 查询参数，会归一化到路径参数） |
| 场景食疗 | `/scenes` | 「校园十大场景」类场景列表 |
| 场景方案 | `/scene/:id` | 某场景下的食疗方案与关联菜谱 |
| AI 生成 | `/ai` | 症状等输入，请求后端大模型能力生成方案（依赖环境配置） |
| AI 方案详情 | `/ai/plan/:id` | 查看某次 AI 方案详情 |
| 我的 | `/profile` | 个人中心入口 |
| 设置 | `/settings` 及子页 | 个人资料、修改密码、账户安全、不感兴趣、隐私政策、饮食偏好等 |

**收藏与浏览历史**等功能由后端 `FavoriteHistory` 等接口支撑，具体 UI 以当前前端实现为准。

## 6. 管理后台功能模块

入口布局为统一后台框架，默认重定向至数据看板。

| 模块 | 路径 | 说明 |
|------|------|------|
| 数据看板 | `/admin/dashboard` | 运营与系统概览类指标（部分内容需具备相应权限的会话） |
| 本周药膳日历 | `/admin/campus-calendar` | 维护面向学生的周历展示（admin / canteen） |
| 药膳管理 | `/admin/recipe` | 菜谱 CRUD 与运营维护（admin / canteen） |
| 食材管理 | `/admin/ingredient` | 食材主数据维护（admin / canteen） |
| 用户管理 | `/admin/user` | 用户启用禁用等（**仅 admin**） |
| 系统设置 | `/admin/system/settings` | 全局参数类配置（**仅 admin**） |
| AI 质量治理 | `/admin/ai-quality` | AI 输出相关治理与配置入口（**仅 admin**） |

场景食疗数据由后端 **场景列表 / 场景菜谱** 等 API 提供（如 `/api/scenes`），与种子数据或运营导入流程配合；详见后端 `SceneTherapyController` 与数据初始化说明。

## 7. 典型使用流程（产品视角）

### 7.1 新生首次使用（学生）

1. 打开应用根路径 → 门户选择 **学生端**。
2. 校园端登录 → 若无体质画像，系统引导 **体质采集**。
3. 完成体质后进入 **首页推荐**，可浏览 **菜谱详情**、**本周日历**、**场景食疗**。
4. 可选：在 **AI 生成** 中输入症状等，获取食疗建议（需后端配置可用的大模型地址与模型名）。
5. 在 **设置** 中维护密码、偏好、隐私与「不感兴趣」等。

### 7.2 食堂运营日常

1. 门户选择 **后台管理** → 使用 **canteen** 类账号登录。
2. 维护 **药膳管理**、**食材管理**，并更新 **本周药膳日历** 以配合学生端展示。

### 7.3 系统管理员

1. 使用 **admin** 账号登录后台。
2. 查看 **数据看板**、处理 **用户管理**（含禁用违规账号）、调整 **系统设置** 与 **AI 质量治理**。
3. 生产环境务必修改默认种子口令、JWT 密钥与数据库凭据，并遵循安全基线（见 `docs/security-baseline-checklist.md` 等）。

## 8. 环境与依赖（摘要）

| 组件 | 说明 |
|------|------|
| 前端 | Node 18+；`tcm-diet-frontend` 下 `npm install` / `npm run dev`；默认 **http://localhost:11999** |
| 后端 | JDK 11；默认 **http://127.0.0.1:11888**；健康检查 **GET `/health`** |
| 数据库 | 开发默认可使用内存 **H2**（无需 MySQL）；连 MySQL 时需配置 profile、账号与初始化策略，详见 `README.md` |
| 大模型 | 可选；通过 `LLM_URL` 或 `LLM_HTTP_HOST` / `LLM_HTTP_PORT` 等环境变量配置；前端开发模式可对 AI 请求使用 mock（`VITE_AI_MOCK=0` 关闭） |

局域网访问后端时需将 **`SERVER_ADDRESS`** 设为 `0.0.0.0` 等，详见 `README.md`。

## 10. 已知产品约束与注意事项

1. **门户不可绕过**：直接打开深链会回到门户或登录前状态，属预期安全与引导行为。
2. **体质冷启动**：无画像时首页会导向体质采集；从场景带标签进入时可例外先浏览场景相关内容。
3. **AI 能力依赖上游**：未正确配置大模型或网络不可达时，AI 相关功能可能失败或降级，需在运维侧监控与告警。
4. **免责声明**：药膳推荐与 AI 生成内容仅供膳食参考，不构成医疗诊断或处方；急重症应就医。
