## 一、开场：用三句话讲清项目

1. **是什么**：面向高校场景的 **Web 应用**，把「中医体质与调养诉求」和 **周更、有限的校园档口供给（药膳日历）** 绑在一起，给出 **可执行、可解释** 的选餐与食疗路径。  
2. **为什么不一样**：不把大模型当成「无限菜谱生成器」，而是让 **周菜单 + 药膳档案** 做「有没有这道菜」的真源，AI 只在 **候选池内** 做说明、排序与组合，并用 **规则优先级** 保证安全与供应一致。  
3. **谁受益**：学生得到 **买得到、说得清** 的建议；学校/后勤获得 **可传播、可管控** 的周更运营与品牌叙事；管理端沉淀 **菜品档案与周历**，降低对模型「编造」的依赖。

## 二、解决的痛点（答辩叙事主线）

| 痛点 | 现实表现 | 本项目的应对 |
|------|----------|----------------|
| **供给与推荐脱节** | 通用养生 App 推荐「某汤某粥」，校园食堂当周根本没有 | 以 **WeeklyMenu（周菜单）已发布数据** 为供给真源；「本周建议点」必须能挂到 **真实 SKU** |
| **AI 幻觉与信任危机** | 模型写出档口不存在的菜，学生到店扑空，易引发舆情 | **AI 不发明当周无货菜品**；职责限于解释、排序、替代说明；可做 **mock/规则拼装降级** |
| **场景碎片化** | 备考、养胃、熬夜恢复等需求分散，缺乏统一入口与固定输出结构 | **十大场景** + **同构模块**（目标句、今日 Top、本周路径、尽量少选、生活配合），并与日历 **同源联动** |
| **安全与禁忌难统一** | 体质、过敏、用药、孕期等若只靠模型自由发挥，风险高 | 明确 **冲突裁决顺序**：安全硬否决 → 供应硬否决 → 忌口 → 不适规避 → 入口意图加权 → 体质调和 |
| **学校侧运营弱** | 周更菜单、停供、主题传播缺少数字化抓手 | 周历 **发布态/未发布/紧急更正** 产品化；为「海报 + 长图 + 台卡」等传播预留 **仅含真实 SKU** 的文案边界 |

答辩时可强调：**痛点不是「没有 AI」，而是「没有与校园供给对齐的可信推荐」**；本项目把问题定义在 **可信落地** 上，而不是炫技。

## 三、创新点（建议分条陈述，每条配一句证据）

1. **「人—货—场」三层对齐的产品模型**  
   - **人**：体质测评、近期不适/调养目标、忌口与偏好等 UserState。  
   - **货**：DishProfile（药膳档案）+ WeeklyMenu（周菜单行项目）。  
   - **场**：十大场景 ScenePack 与日历绑定策略。  
   - **证据**：业务上坚持「本周可点 ⊆ 已发布周历」；工程上见推荐域服务（如 `RecommendApplicationService` / 过滤与打分域服务）与菜谱查询层对周历、标签的约束。

2. **AI 职责边界清晰：有限池内「讲清楚、排好序、说到做到」**  
   - 允许：理由、组合、替换说明、传播短文案（且须校验在池内）。  
   - 禁止：把不存在于本周菜单的 SKU 写进「本周建议点」。  
   - **证据**：**`campus.ai.mock`**（为 true 时不调上游、走本地拼装）、顶层 **`llm.*`** 与环境变量 **`LLM_*`**（网关地址、模型、超时）；系统表 / KV **`ai.generation.enabled`**（管理端「系统设置」可关，关闭后接口直接返回「功能已关闭」）；《大模型调用调试说明》；`AiDietService` / `AiTherapyPlanService` 等对未配置上游或上游失败时的兜底路径。食疗侧还可加载 **`classpath:llm-skills/runtime-llm/...`** 片段（`RuntimeLlmSkillLoader` 等），把合规与输出格式从提示词层**模块化**，便于迭代与审计。

3. **规则与数据优先于模型「直觉」**  
   - 推荐权重（体质/季节匹配 vs 热度）、体质九题量表映射、场景与周历 API 等，体现 **可审计、可复现** 的业务逻辑。  
   - **证据**：配置项 **`campus.recommend.match-weight` / `popularity-weight`**（以 `application.yml` 为准）；`test_integration.py` 对登录、体质、推荐、AI、收藏、场景、管理端等路径的覆盖（Python 标准库即可跑）；后端另有 **JUnit** 单测与 **`scripts/Run-BackendTests.ps1`**（含安全基线相关用例）。

4. **合规与体验双设计**  
   - 固定免责声明、功效表述规范、**「AI 生成区」与「本周可点区」视觉分区**（产品要求），降低误读与合规风险。

5. **工程化交付完整**  
   - 前后端分离、JWT 鉴权、校园端与管理端分域路由、启动期种子数据（演示菜谱 / 交互 / 周历，可在生产评估关闭）、Windows 下一键脚本（`启动后端.bat`、`启动前端.bat`、可选 MySQL 开发实例）、**`test_integration.py`**；前端另含 **`npm run test:e2e`（Playwright）** 与 CI 门禁脚本（如 **`npm run test:e2e:ci`**，见 `README.md`）；仓库含 **GitHub Actions**（后端测试、前端构建与供应链审计等，见 `.github/workflows/`）。

## 四、项目核心优点（评委常问「你的优势是什么」）

1. **可信**：推荐与 **实物菜单** 对齐，从根上减少「推荐落不了地」的体验与品牌风险。  
2. **可解释**：不是黑盒分数，而是能说出 **为什么推荐 / 为什么排除某类原因**（与裁决顺序一致）。  
3. **可运营**：周更节奏、停供更正、场景包与菜品标签，把 **中医顾问与后勤** 的工作固化为数据与流程。  
4. **可降级**：大模型不可用或内网不通时，仍可依赖 **规则与模板** 维持核心路径演示与基本服务。  
5. **可交付**：文档链（**README、AGENTS、路径清单、大模型说明**等）+ **`test_integration.py`** 与 CI，**可复现、可移交**。

## 五、系统提供的服务与能力（按角色组织，便于答辩「功能清单」）

### 5.1 学生（校园端）

- **门户入口**（`/`）：先选「学生端」或「后台管理」，再进入对应登录；避免直接粘贴登录 URL 绕过门户（路由守卫与答辩演示话术可一带而过）。  
- 校园端注册登录、**个人中心 / 设置**（个人资料、修改密码、账户安全、饮食偏好、不感兴趣、隐私政策等）  
- **体质测评**与档案（九题量表 → 体质倾向）；无档案时进入首页会引导先完成测评（冷启动引导；从场景带 `scene_tag` 进入时可先浏览再引导）  
- **本周校园药膳日历**（`/calendar`；按日、餐段、食堂等维度浏览；联动菜谱详情）  
- **十大场景**（`/scenes` → `/scene/:id`）：同构模块（目标句、今日 Top、本周路径、尽量少选、生活配合等），与周历同源解析逻辑

- **AI 食疗**（`/ai`、方案详情页；在 **`ai.generation.enabled` 开启**、**`campus.ai.mock` 为 false** 且网关 URL/模型有效时走上游；否则本地拼装或返回「功能已关闭」）

### 5.2 管理员

- 管理端登录、**数据看板**（可结合 **`GET /api/admin/dashboard/runtime-metrics`**、**`/observability-summary`** 等谈「本地可观测」，字段说明见 `docs/observability-metrics.md`）  
- **药膳库、食材**、**用户管理**（部分菜单按角色展示）  
- **校园周历**维护（与「供给真源」直接相关）  
- **`admin` / `canteen` 角色**：食堂运营可维护周历与菜谱相关页；仅 **`admin`** 可进用户管理与系统设置（与前端 `adminRoles` 一致）

### 5.3 技术与运维侧（可作为「工程质量」表述）

- REST API 模块化（认证、用户档案、校园推荐、场景、AI、管理端等）

- `test_integration.py` 覆盖多路径；后端 **`mvnw` + JUnit`**；前端 Playwright E2E 与 CI

## 六、项目闪光点（答辩时「记忆点」话术）

1. **问题定义准**：不做「泛养生百科」，而做 **校园限定供给下的决策辅助**。  
2. **不变量清晰**：凡标注「本周可点」的条目，满足 **∈ 已发布周菜单 ∩ 未被安全/供应/忌口规则否决** —— 这是可对外承诺的产品纪律。  
3. **AI 定位克制**：AI 是 **讲解员与排序员**，不是 **进货单捏造者**；体现对生成式 AI 风险的清醒认识。  
4. **分期路线务实**：一期强调周历 + 场景同构 + 不变量校验；二期 intersection、停供替换与订阅；三期传播物料与轻预约 —— 体现 **迭代思维** 而非一次性空想。  
5. **全栈闭环**：Vue 3 + Vite 6 + Spring Boot 2.7 + MySQL + OpenAI 兼容网关 + 联调 / E2E + CI，**从需求到本地可演示部署** 可讲完整故事。

## 七、适合的应用场景（「在哪些学校/部门用」）

| 场景 | 说明 |
|------|------|
| **高校后勤与餐饮服务中心** | 已有或计划推出「药膳窗口 / 主题周」，需要线上载体统一发布周菜单并引流。 |
| **学生工作 / 健康教育** | 结合体质关怀、考试季、季节养生等开展校园活动，需要 **可打卡、可解释** 的数字化工具。 |
| **中医药学院或校医院健康教育（非诊疗）** | 强调「膳食调养不替代诊疗」，用于科普与生活方式指导的 **配套信息化**。 |
| **智慧校园试点** | 作为餐饮与文化板块的子系统，与统一身份认证、消息推送等 **后续对接**（可作为论文「展望」）。 |
| **课程设计 / 毕业设计答辩** | 技术栈主流、业务边界清楚、文档齐全，适合展示 **产品思维 + 工程实现 + AI 治理** 综合能力。 |

## 八、技术架构简述（30 秒版，背熟即可）

- **前端**：`tcm-diet-frontend`（Vue 3 + **Vite 6**；Element Plus / Vant；Pinia 持久化；开发服务默认 **`http://localhost:11999`**；开发态 **`/api` → Vite 代理** 到 **`VITE_PROXY_TARGET` 或 `http://127.0.0.1:11888`**；**使用系统 PATH 中的 Node/npm**，建议 **Node 18+**）。  
- **后端**：`campus-diet-backend`（**Spring Boot 2.7.18**、**Java 11**——以 `pom.xml` 的 **`java.version`** 为准；本仓库脚本与 CI 均按 **JDK 11** 约定；可用 **`mvnw.cmd`** + **`maven-settings-d.xml`** 降低本机 Maven 版本差异）。技术栈含 MyBatis-Plus、JWT（jjwt）等。  
- **数据**：MySQL 库 **`tcm_diet`**；连接串可用 **`MYSQL_*`** 或 **`SPRING_DATASOURCE_URL`** 覆盖；启动 SQL 见 `spring.sql.init`（已有库生产环境常用 **`SPRING_SQL_INIT_MODE=never`**）。  
- **服务端口与监听**：默认 **`11888`**，**默认仅监听 `127.0.0.1`**（`SERVER_ADDRESS` 未设置时）；需手机或同网其它机器访问后端时，设 **`SERVER_ADDRESS=0.0.0.0`** 并配合防火墙与 CORS。探活 **`/health`**、**`/api/health`**。  
- **大模型**：OpenAI 兼容 **Chat Completions**；默认 URL 由 **`LLM_HTTP_HOST` / `LLM_HTTP_PORT`** 等与顶层 **`llm.url`** 拼接（也可用 **`LLM_URL`** 覆盖整段）；本机 Ollama 等可配 **`LLM_MODEL`**（**`LLM_API_KEY` 可空**）。**Mock**：**`campus.ai.mock`**；**业务总开关**：系统 KV **`ai.generation.enabled`**。详情见 **`llm.*`**、`campus.ai.*` 与《大模型调用调试说明》。  
- **质量保障**：根目录 **`test_integration.py`**（Python 3.8+，标准库）；后端 **`./mvnw test`** 或 **`scripts/Run-BackendTests.ps1`**；前端 **`npm run test` / `test:e2e` / `test:e2e:ci`**（见 `README.md`）。

## 九、局限与诚实表述（评委加分项：体现批判性思维）

- **不替代医疗**：食疗建议不构成诊疗；过敏与禁忌以用户自述与医嘱为准。  
- **生产化待加强**：如统一身份认证、一卡通、高可用集群、独立迁移工具链（Flyway/Liquibase）等，可在「展望」中陈述；当前已有 **`ProdSecurityBaselineValidator`** 等基线，不等于已完成全部运维面。  
- **依赖校方数据质量**：周菜单及时发布与停供更正，决定推荐上限；系统是 **放大器** 而非 **替代后勤执行**。  
- **大模型稳定性**：依赖网关 SLA；后端已配置读取超时（`llm.timeout` / `LLM_TIMEOUT_MS`）；可将 **`campus.ai.mock`** 设为 **true** 或在管理端关闭 **`ai.generation.enabled`**，演示与弱网环境仍可走本地拼装或明确提示；生产仍需监控与熔断策略深化。  
- **演示密钥**：`application.yml` 中带默认网关示例密钥，答辩可说明**生产必须用环境变量注入并轮换**，勿当作真实密钥展示。

## 十、答辩常见问题与回答思路（简答提纲）

**Q：你们和市面上养生 App 有什么区别？**  
A：区别在于 **供给模型**。我们假设 SKU 是 **周更且有限** 的，所有「本周可点」输出必须与周菜单交集；通用 App 往往不具备这一真源，容易「说得对但买不到」。

**Q：AI 在你们系统里到底起什么作用？**  
A：在 **候选集合非空** 时，负责自然语言解释、排序理由、组合与替代说明；**不负责**判定当周有没有某道菜——后者由数据库中的周历与供应状态决定。

**Q：如何保证安全？**  
A：产品层有 **裁决优先级**（安全与供应优先于偏好与体质调和）；工程层有过滤与档案标签；展示层有免责声明与分区，避免用户混淆「建议」与「诊疗」。

**Q：数据从哪来？**  
A：运营/管理员维护 **药膳档案与周菜单**；学生侧产生 **体质与偏好**；系统计算交集与排序；AI 在交集之上增强可读性。

**Q：大模型挂了怎么办？**  
A：将 **`campus.ai.mock`** 设为 **`true`**（不调上游、本地拼装），或在管理端关闭 **`ai.generation.enabled`**（用户侧看到「功能已关闭」而非超时）；推荐流与周历仍来自规则与数据库，核心路径可演示。

**Q：创新点一句话？**  
A：**把「可信落地」写进产品不变量**：校园药膳推荐必须能挂到本周真实 SKU，AI 在池内增强表达而非虚构供给。

**Q：为什么有个「选模式」首页，不能直接打开登录页？**  
A：产品上将 **学生端与后台** 分流，并在路由层要求 **从门户入口进入**，减少误闯后台、也便于答辩时一句话讲清 **双端交付**。

## 十一、演示建议（若答辩含现场演示）

1. **环境**：MySQL 已启动且库 **`tcm_diet`** 可用 → 后端 **`http://127.0.0.1:11888/health`** 或 **`http://127.0.0.1:11888/api/health`** 返回正常 → 再开前端 **`http://localhost:11999`**（根目录 **`启动后端.bat` / `启动前端.bat`** 或 `README.md` 手动命令）。  
2. **顺序**：健康检查 → 浏览器打开 **`/`** → 选 **学生端** → 校园登录 →（新用户会先 **体质测评**）→ **首页推荐** → **本周药膳日历** → **场景食疗** 任一场景 →（可选）**/ai** → 回到 **`/`** 选 **后台管理** → 管理登录 → **周历 / 药膳管理**。  
3. **话术配合**：每到「本周可点」列表，口头对应 **不变量**；到 AI 区，强调 **分区与边界**。  
4. **风险预案**：现场无内网 LLM 时，提前设 **`campus.ai.mock: true`**、或配置本机 **Ollama**（`LLM_URL` / `LLM_MODEL`）、或在管理端关闭 AI 总开关，避免冷场。

本项目面向 **校园周更餐饮现实**，用 **「人—货—场」** 统一建模，把 **周菜单与药膳档案** 作为可信真源，把 **大模型** 约束在 **有限候选池内的解释与排序**，并配套 **门户双端、管理端、文档与联调 / E2E**，形成可交付、可演进、边界清晰的 **校园药膳推荐系统**。适合作为 **智慧后勤、健康教育、中医药文化进校园** 等方向的数字化载体；后续可在统一身份认证、消息通知与供应链联动上继续深化。
