# 项目优化方向建议

## 一、现状判断（综合，2026-04-16）

项目已具备「可运行、可联调、可门禁验证」的形态（前端 Vue 3 + Vite，后端 Spring Boot + MyBatis）。**与早期静态扫描结论相比**，工程化与安全护栏已显著补齐；当前主要矛盾转为**可持续重构节奏**与**局部技术债消化**，表现为：

- **安全**：`prod` 由 `ProdSecurityBaselineValidator` 阻断弱 JWT、空/弱 LLM 密钥、演示种子开关、`spring.sql.init.mode=always`、默认种子口令等；CI 与 PowerShell 双通道执行 `check-prod-security-baseline` 静态断言。**本轮补强**：生产环境额外拒绝 `application.yml` 中的数据库口令占位默认值 `change-this-db-password` 及少量常见弱口令，避免「只改 profile 未改 MYSQL_PASSWORD」的误上线。
- **质量门禁**：`.github/workflows/ci.yml` 已覆盖前端 audit（critical）/lint/typecheck/test/Playwright/build/E2E、dist 与构建时长门禁；后端 `./mvnw test` + 安全基线脚本；工作流显式 `permissions` + `concurrency`（最小权限 + 同 ref 取消在途运行）。
- **测试**：后端 `src/test` 已覆盖认证、推荐、周历、管理端契约、安全过滤器、生产基线等（数量见第七节本轮证据）；前端具备单测与 CI 内 E2E。
- **技术债（非阻断）**：部分核心页面/服务仍偏大；UI 双栈与 JS 存量按 P1 渐进治理即可。

---

## 二、优化方向（按优先级）

### P0（立即推进）

1. **安全与稳定性基线**
   - 移除默认密钥/默认口令
   - 敏感配置迁移到环境变量或安全配置中心
   - 增加关键异常日志与基础审计日志

2. **质量门禁建设**
   - 建立最小 CI 流程：安装依赖 -> 构建 -> 测试 -> 失败阻断
   - 前端补齐 `lint` / `typecheck` / `test` 门槛
   - 后端补齐 `test` 门槛，形成提交即验证机制

3. **后端测试从 0 到 1**
   - 优先覆盖高风险链路：认证、推荐、周历等关键接口
   - 至少建设 Controller + Service 的最小集成测试

### P1（短中期）

4. **架构可维护性提升**
   - 拆分超大 Vue 页面与后端长服务类
   - 引入清晰的分层边界（业务编排层 / 领域服务 / 基础设施层）

5. **前端一致性治理**
   - 统一 UI 主栈（或明确分域使用策略）
   - 逐步推进 JS -> TS 收敛
   - 统一 API 层规范，降低重复实现与边界混乱

### P2（持续优化）

6. **性能与构建体验**
   - 重依赖按需加载与懒加载
   - 优化构建 chunk 策略
   - 建立包体积与构建时长监控，防止性能回退

---

## 三、方案对比（供决策）

### 方案 A：稳健增量（推荐）

**特点**：先补安全、测试、CI，再逐步重构。  

**优点**
- 风险低，不中断当前功能迭代
- 2~4 周可见明显质量收益
- 组织协同成本较低

**缺点**
- 历史结构问题改善速度相对慢
- 技术债消化周期更长

**适用场景**
- 当前仍有持续功能需求
- 团队资源有限，需平衡交付与治理

### 方案 B：结构化重整（激进）

**特点**：集中窗口做模块拆分、架构收敛、规范统一。  

**优点**
- 中长期维护成本显著下降
- 架构更清晰，测试与扩展更容易铺开

**缺点**
- 短期投入较大
- 回归风险与沟通成本更高

**适用场景**
- 近期功能压力较小
- 可以投入专门重构周期

---

## 四、30 天可执行路线（基于方案 B）

### 第 1 周：重整准备与基线兜底
- 完成前后端模块盘点，明确拆分优先级与依赖关系
- 冻结高风险需求变更窗口，建立重整期间分支与发布策略
- 同步落地 P0 底线项（默认密钥/口令清理、最小 CI 骨架）

### 第 2 周：核心模块拆分与边界重建
- 拆分 1~2 个最大前端页面，沉淀共享组件与状态边界
- 拆分 1~2 个后端长服务类，建立业务编排层与领域服务边界
- 统一 API 层输入输出规范，减少跨模块耦合

### 第 3 周：技术栈收敛与迁移验证
- 明确并执行 UI 主栈收敛策略（保留主栈、限制次栈新增）
- 对核心高变更模块推进 JS -> TS 收敛
- 为拆分后的关键链路补齐最小集成测试并接入 CI

### 第 4 周：回归稳定与发布切换
- 执行全链路回归与性能基线对比，修复拆分引入的问题
- 推进重依赖懒加载与构建 chunk 优化，验证包体收益
- 制定并演练灰度发布与回滚预案，完成阶段性重整验收

## 五、风险与注意事项

- **性能结论**：仍以业务压测与前端 Lighthouse/包体 artifact 为校准来源；CI 内 dist/构建时长阈值为回归护栏，非容量规划结论。
- **重构**：大规模拆分须以「测试/契约为先」；当前仓库已具备基础护栏，剩余风险主要来自**改动范围**而非「从零单测」。
- **环境差异**：非 GitHub 的发布流水线需自行接入与 CI 同等的安全基线脚本；Windows 无 Bash 时以 `check-prod-security-baseline.ps1` 或 CI 为准。
- **方案 B**：短期仍可能压缩功能窗口，需人工冻结范围（见第六节）。
- **架构**：拆分若缺少边界设计，可能出现「耦合转移」；后端推荐/食疗等已提供分层参考实现。

## 六、建议的下一步

1. **（仅人工）** 确认本次重整窗口与冻结范围（哪些需求延期、哪些问题必须并行修复）。
2. **（仅人工）** 将每周重整目标拆成 issue 级任务（负责人、截止时间、验收标准、回滚点）。
3. **（已自动化）** 坚持「护栏先行再切换」：仓库 **CI** 已覆盖前端 `lint` / `typecheck` / `test` / `build` / E2E 门禁与后端 `mvn test` + 生产安全基线脚本；**prod 库口令**除禁止 `root` 外，已对齐拒绝 `application.yml` 占位默认值与常见弱口令（见 `ProdSecurityBaselineValidator`）；拆分与重构仍以 **本地/PR 全绿** 为进入下一批的硬门槛。

> 更细的「路径 2」剩余人工项（推送远程 CI、prod 进程级演练等）见 **`路径2-未完成项清单.md`**，与本节互补而非重复。
