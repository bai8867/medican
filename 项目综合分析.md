# 校园药膳推荐项目综合分析

## 1. 项目定位与目标

本项目面向校园日常食养场景，构建了“学生端推荐与食疗建议 + 后台运营管理”的一体化 Web 系统。  
当前实现目标不是“医疗诊断系统”，而是“可执行的校园食养辅助平台”：在保障基础可用和权限边界的前提下，为学生提供推荐、收藏、浏览、AI 方案等闭环能力，并为管理侧提供内容供给和策略控制能力。

从工程结构看，项目由两个核心子系统组成：

- `tcm-diet-frontend`：Vue 3 前端工程，统一承载学生端和后台端界面。
- `campus-diet-backend`：Spring Boot 后端工程，提供鉴权、推荐、AI、收藏历史、后台管理和系统配置 API。

---

## 2. 需求分析

## 2.1 业务需求

- 学生侧支持按季节、体质、场景、关键词浏览药膳推荐流。
- 支持菜谱详情、收藏与浏览历史，形成可回流的行为链路。
- 支持 AI 食疗方案生成，输出结构化字段和可直接展示的 Markdown 文案。
- 支持后台角色进行药膳、食材、周历、用户和系统开关管理。

## 2.2 用户角色与权限需求

- 学生用户（`USER`）：
  - 校园端登录、体质采集、推荐浏览、收藏历史、AI 方案生成。
- 食堂管理（`CANTEEN_MANAGER`）：
  - 可访问运营子集模块（看板、周历、药膳、食材）。
- 管理员（`ADMIN`）：
  - 可访问全部后台能力（含用户管理、系统设置、策略开关）。

权限控制是前后端双层实现：

- 前端路由基于 `beforeEach` 做登录校验、门户入口校验、角色校验、禁用状态跳转。
- 后端接口通过登录态解析和 `SecurityUtils` 角色限制执行最终访问控制。

## 2.3 核心功能需求拆解

- 账号体系：注册/登录、JWT 会话、禁用用户拦截（业务码 `4031`）。
- 推荐体系：候选过滤、综合排序、分页输出、可配置权重、可开关。
- 菜谱体系：详情展示、收藏状态联动、管理端 CRUD、状态维护。
- AI 体系：OpenAI 兼容接口调用 + 解析校验 + 本地兜底 + Markdown 输出。
- 运营体系：后台模块化管理 + 全局策略开关 + 系统 KV 参数管理。

## 2.4 非功能需求

- 可用性：推荐和 AI 均支持系统开关，AI 失败可降级为本地兜底方案。
- 可维护性：前后端按业务域拆分；配置集中在 `application.yml` 与环境变量。
- 可扩展性：推荐算法路径、权重、候选窗口、系统开关均可运行时调整。
- 可联调性：前端支持 `/api` 代理和 Mock 网关混合模式，降低后端不可用时阻塞。

## 3. 技术实现分析

## 3.1 整体架构

项目采用前后端分离架构，当前技术栈如下：

- 前端：`Vue 3 + Vite 6 + Pinia + Vue Router + Axios + Element Plus + Vant + ECharts`
- 后端：`Spring Boot 2.7.18 + MyBatis-Plus + MySQL + JWT`
- AI：OpenAI 兼容 Chat Completions 接口（可接自定义网关或本地模型网关）

运行层面：

- 前端开发端口为 `11999`，默认将 `/api` 代理到 `http://127.0.0.1:11888`。
- 后端默认端口为 `11888`，**默认仅监听 `127.0.0.1`**（与 `application.yml` 中 `server.address` 一致）；若需局域网或容器宿主机从外部访问，设置环境变量 **`SERVER_ADDRESS=0.0.0.0`** 覆盖。

## 3.2 前端实现要点

- 路由与门禁：
  - 统一路由同时承载学生端与后台端。
  - 校园端与后台端都引入“门户入口校验”，避免直接粘贴内部路径绕过入口流程。
  - 对后台子路由应用角色约束：`admin` 与 `canteen` 的可访问范围不同。
- 状态管理：
  - `Pinia + pinia-plugin-persistedstate` 持久化校园端和后台端会话与偏好状态。
  - 管理端角色由后端 `ADMIN/CANTEEN_MANAGER` 映射为前端 `admin/canteen`。
- 网络与异常处理：
  - 请求层按路径自动注入对应 token（校园端与后台端分键存储）。
  - 对账号禁用业务码 `4031` 做统一会话清理与跳转。
  - 引入 Mock 网关，实现真实接口与 Mock 的按需切换。

## 3.3 后端实现要点

- 业务分层：
  - 控制器按领域拆分为鉴权、推荐、场景、AI、收藏历史、后台管理等模块。
  - 服务层承载核心业务逻辑，Mapper 层基于 MyBatis-Plus 实现数据访问。
- 鉴权与安全：
  - `AuthService` 限制注册角色，仅允许 `USER` 与 `CANTEEN_MANAGER`，禁止自助注册 `ADMIN`。
  - `JwtService + JwtAuthFilter` 完成身份解析；用户禁用状态在登录与访问阶段双重校验。
  - 生产环境存在安全基线校验（如 JWT 秘钥强度、AI 密钥要求）。
- 推荐逻辑（`RecipeRecommendService`）：
  - 支持关键词、场景、季节、体质等多维筛选。
  - 评分由匹配分、热度分组成；在 `hybrid` 路径下可叠加行为加权（收藏/浏览）。
  - 权重项 `match-weight`、`popularity-weight`、`behavior-weight` 可配置。
  - 全局受 `recommend.global.enabled` 控制，关闭时直接返回空结果。
- AI 逻辑（`AiTherapyPlanService`）：
  - 优先调用上游大模型，失败时回退到本地方案拼装。
  - 对模型输出做结构化合并与字段补全，避免前端渲染不稳定。
  - 统一返回 `therapyRecommendMarkdown`，保证页面可直接消费。
  - 全局受 `ai.generation.enabled` 控制，关闭时返回明确提示结构。

## 3.4 数据与配置

- 数据库默认 MySQL，启动可通过 `schema.sql + data.sql` 初始化。
- 系统启动时支持注入演示数据（药膳、交互、周历）以提升开发联调效率。
- 配置支持环境变量覆盖，核心包括：
  - 数据库连接（`MYSQL_*`）
  - LLM 地址、模型、密钥、超时（`LLM_*`）
  - JWT 秘钥（`CAMPUS_JWT_SECRET`）
- 系统策略支持后台动态调整：
  - `recommend.global.enabled`
  - `ai.generation.enabled`
  - 通用 `system_kv` 键值配置

## 3.5 测试与运行

- 前端：
  - 单测入口：`npm run test`
  - E2E 入口：`npm run test:e2e`（Playwright）
  - 已存在学生端与跨端导航相关 E2E 用例。
- 后端：
  - 已包含推荐服务、AI 服务、安全工具等单测类。
  - 可通过 `mvn spring-boot:run` 启动本地服务。
- 工程脚本：
  - 根目录提供 MySQL/后端/前端启动脚本，降低本地环境准备成本。

## 4. 项目逻辑（业务闭环）

## 4.1 学生端主链路

1. 用户从入口页选择校园端，进入校园登录流程。
2. 登录后系统检查画像完整性；缺少体质信息时引导进入体质采集页。
3. 用户在推荐页按季节/体质/场景/关键词浏览推荐流并查看详情。
4. 用户进行收藏与浏览，行为写回后端形成偏好数据。
5. 用户在 AI 页面输入症状，获得结构化食疗建议与 Markdown 方案。
6. 用户在设置页维护个人资料与偏好，反向影响后续推荐上下文。

该链路已形成“画像 -> 推荐 -> 行为 -> 再推荐”的可持续循环。

## 4.2 推荐逻辑主线

- 输入：分页参数 + 场景标签 + 体质编码 + 是否个性化 + 季节 + 关键词。
- 过滤：候选池先按状态、关键词、场景等条件筛选。
- 打分：
  - 热度分：基于收藏量对数归一化；
  - 匹配分：季节匹配 + 体质匹配；
  - 行为加权（可选）：在 `hybrid` 模式下叠加收藏/浏览偏好。
- 输出：按综合分排序后分页返回，同时包含 `hasMore` 和推荐理由。

该实现兼顾规则可解释性与个性化空间，适合校园场景的渐进式演进。

## 4.3 AI 方案生成逻辑

- 用户提交症状（可带体质）后，后端组装提示词并调用 OpenAI 兼容接口。
- 模型返回后进行 JSON 解析、字段补全、菜谱映射校验和去重处理。
- 当上游异常、配置缺失或功能关闭时，系统返回可读的本地兜底结果。
- 最终输出包含：
  - 菜谱推荐清单
  - 关键食材建议
  - 生活方式建议
  - 禁忌提醒
  - `therapyRecommendMarkdown` 文案

核心目标是“在上游不稳定情况下仍可用”，避免 AI 成为单点故障。

## 4.4 后台运营逻辑

- 后台采用分权访问：
  - `ADMIN`：全量管理
  - `CANTEEN_MANAGER`：运营子集管理
- 运营模块围绕三类能力展开：
  - 内容供给：药膳、食材、周历维护
  - 用户治理：用户查询与状态管理
  - 策略治理：推荐/AI 开关、系统 KV 参数
- 看板支持基础业务指标与运行时指标查看，便于运营和调优联动。

该设计将策略与内容从代码中抽离，逐步向“可运营系统”靠拢。

## 5. 项目特点与改进建议

## 5.1 当前特点

- 前后端边界清晰，学生端与后台端在同一前端工程下实现双门户治理。
- 推荐与 AI 均支持开关和兜底，业务连续性较好。
- 权限控制覆盖路由层与接口层，且禁用用户处理较完整。
- 已具备基础可观测能力（运行计数与平均耗时快照接口）。

## 5.2 可优先改进方向

- 可观测性增强：
  - 将运行时指标对接持久化监控系统（如 Prometheus/Grafana），补齐时序留存与告警（落地步骤见 **5.3**）。
- 推荐效果评估：
  - 引入离线评估指标（点击率、收藏率、转化率）与 A/B 开关，验证权重配置有效性。
- AI 质量治理：
  - 增加输出质量评分与安全规则校验，沉淀“可回放问题样本集”。
- 测试深度提升：
  - 补齐推荐边界、权限越权、系统开关联动等关键路径集成测试。
- 数据闭环优化：
  - 将收藏/浏览行为从“同菜谱加权”升级为“标签级兴趣画像”。

### 5.3 可观测性实施方案（方案 A：Prometheus + Alertmanager + Grafana）

本方案以「最小可用、先跑通闭环」为目标：指标进入 Prometheus 长期可查，Alertmanager 负责路由与降噪，Grafana 负责可视化与值班入口。与引入 Thanos/Mimir 等组件相比，部署与心智成本更低，适合作为第一阶段基线；若后续单机磁盘或查询压力成为瓶颈，再评估水平扩展方案。

仓库内已落地编排与初版告警规则：见 **`observability/`**（`docker compose` 启动三件套；后端暴露 **`/actuator/prometheus`**，抓取目标默认 `host.docker.internal:11888`，详见该目录下说明）。

**1）指标采集（应用侧）**

- 后端（Spring Boot）：接入 Micrometer Registry for Prometheus，暴露标准 `/actuator/prometheus`（或统一网关下的等价路径）；确保该端点**不经过业务 JWT** 或单独使用 Basic/网络 ACL，避免监控抓取被鉴权拦截。
- 指标分层建议：
  - **RED**：请求速率、错误率、延迟分位数（HTTP 服务端）。
  - **资源**：JVM 内存/GC、线程池、数据库连接池（若已使用 HikariCP 等可自带指标）。
  - **业务**：推荐接口 QPS、AI 调用成功率/超时次数、关键开关状态（可与现有「运行计数与平均耗时」能力对齐或迁移为 Counter/Histogram）。
- 前端：静态站点通常不直接暴露 Prometheus 指标；若需体验类指标，可通过 RUM 或网关访问日志间接观测，本阶段可不作为硬要求。

**2）Prometheus（存储与抓取）**

- 使用静态 `scrape_configs` 或 `file_sd_configs` 声明后端实例；开发环境可用 `127.0.0.1:11888`，生产按实际服务发现（K8s `kubernetes_sd`、Consul 等）替换。
- **时序留存**：通过 `--storage.tsdb.retention.time`（例如 `15d` 或 `30d`）与磁盘容量规划匹配；告警与排障以近两周数据为主时，`15d` 往往足够。
- **抓取间隔**：默认 `15s` 或 `30s`；过高频率会增加序列基数与磁盘写入，需与告警灵敏度权衡。

**3）告警（Alertmanager）**

- 在 Prometheus 中编写 `recording rules`（可选，用于复杂查询预计算）与 `alerting rules`，典型初版规则包括：
  - 实例不可达或连续抓取失败；
  - HTTP 5xx 比例或错误率突增；
  - P95/P99 延迟超阈值且持续一定时间（避免单次抖动误报）；
  - JVM 堆使用过高或 Full GC 频率异常；
  - 外部依赖（如 LLM）超时/失败率升高（若已有对应指标）。
- Alertmanager 配置**路由与接收器**（邮件、钉钉/飞书 Webhook 等），并启用**分组、抑制、静默**：同一根因的多条告警合并通知，发布窗口内静默预期抖动。

**4）可视化（Grafana）**

- 将 Prometheus 配置为默认数据源；导入社区 **JVM / Spring Boot** 面板模板并裁剪为本服务维度。
- 建议至少两张看板：**服务健康（RED + 资源）**、**业务与依赖（推荐/AI/开关）**；告警面板与数据源绑定，便于从通知跳转复盘。

**5）验证与上线检查清单**

- 抓取探活：`Prometheus -> Status -> Targets` 中后端 job 为 `UP`。
- 查询抽查：在 Grafana Explore 中能查到核心 Histogram/Summary 分位数或等价聚合。
- 告警演练：人为制造短时错误峰值或停实例，确认 Alertmanager 收到一条**可读、可定位**的通知，且恢复后能收到 resolved（若渠道支持）。
- 安全：指标端点不对外网开放；生产秘钥与告警 Webhook URL 走环境变量或密钥管理，不入库明文。

## 6. 结论

当前项目已经具备“校园食养推荐平台”的可运行雏形：

- 在功能层面，学生端与管理端形成了从内容供给到用户消费的基本闭环；
- 在技术层面，推荐与 AI 均具备可配置、可降级、可扩展能力；
- 在工程层面，模块划分、配置治理和基础测试入口已较完善。

下一阶段建议围绕“监控可视化、推荐效果量化、AI 输出稳定性”持续建设，推动项目从课程型实现向可持续运营系统演进。
